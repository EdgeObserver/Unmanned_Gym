# 预约模块设计说明文档

## 1. 概述

预约模块是无人健身房管理系统的核心功能之一，主要负责处理用户对教练和器材的预约请求。该模块采用了微服务架构，独立部署为BookService，通过REST API提供预约相关功能。

## 2. 模块架构

### 2.1 技术栈
- **框架**: Spring Boot + MyBatis-Plus
- **数据库**: MySQL
- **通信**: OpenFeign (用于服务间调用)
- **数据传输**: DTO模式

### 2.2 分层架构
```
┌─────────────────┐
│   Controller    │  ← REST API接口层
├─────────────────┤
│    Service      │  ← 业务逻辑层
├─────────────────┤
│     Mapper      │  ← 数据访问层
├─────────────────┤
│   Database      │  ← 数据存储层
└─────────────────┘
```

## 3. 核心类说明

### 3.1 控制器层 (Controller Layer)

#### BookController
- **职责**: 处理HTTP请求，协调服务层完成预约操作
- **主要方法**:
  - `bookCoach()`: 处理教练预约请求
  - `bookEquipment()`: 处理器材预约请求
  - `cancelCoachBooking()`: 取消教练预约
  - `cancelEquipmentBooking()`: 取消器材预约
  - `getMyCoachBookings()`: 获取用户教练预约记录
  - `getMyEquipmentBookings()`: 获取用户器材预约记录

### 3.2 服务层 (Service Layer)

#### CoachBookService & CoachBookServiceImpl
- **职责**: 处理教练预约相关的业务逻辑
- **核心功能**:
  - 验证预约时间有效性
  - 检查时间段冲突
  - 创建/更新预约记录

#### EquipmentBookService & EquipmentBookServiceImpl
- **职责**: 处理器材预约相关的业务逻辑
- **核心功能**:
  - 验证预约时间有效性
  - 检查器材可用性
  - 验证每日预约限制(最多2个)
  - 检查是否需要教练指导
  - 创建/更新预约记录

### 3.3 数据访问层 (Mapper Layer)

#### CoachBookingMapper
- 继承MyBatis-Plus的BaseMapper
- 提供教练预约记录的CRUD操作

#### EquipmentBookingMapper
- 继承MyBatis-Plus的BaseMapper
- 提供器材预约记录的CRUD操作

## 4. 业务流程

### 4.1 教练预约流程
1. 用户选择教练、日期和时间段
2. 系统验证日期是否有效(不能是过去日期)
3. 检查该时间段是否已被预约
4. 如果可用，创建预约记录并返回成功信息
5. 如果不可用，返回相应的错误提示

### 4.2 器材预约流程
1. 用户选择器材、日期和时间段
2. 系统验证日期是否有效
3. 检查器材是否存在
4. 验证用户当天预约数量(不超过2个)
5. 检查该时间段器材剩余数量
6. 如果器材需要教练指导，验证用户是否已预约同一时间段的教练
7. 所有验证通过后，创建预约记录

### 4.3 取消预约流程
1. 用户请求取消预约
2. 系统验证预约记录是否存在
3. 验证用户是否有权限取消(只能取消自己的预约)
4. 验证预约日期是否已过期(不能取消过去的预约)
5. 将预约状态更新为已取消(status=0)

### 4.4 查询预约记录流程
1. 用户请求查看预约记录
2. 根据用户ID查询相关预约记录
3. 过滤掉已删除的记录
4. 按日期降序排列返回结果

## 5. 业务规则

### 5.1 通用规则
- 不能预约过去的日期
- 不能取消已过期的预约
- 只能取消自己的预约记录
- 预约采用软删除机制(is_deleted字段)

### 5.2 教练预约规则
- 每个时间段只能被一个用户预约
- 预约成功后立即占用该时间段

### 5.3 器材预约规则
- 每天最多预约2个器材
- 同一用户在同一时间段不能重复预约同一器材
- 部分器材需要教练指导才能使用
- 预约数量不能超过器材总数量

## 6. 异常处理

### 6.1 常见异常情况
- 日期无效: "不能预约过去的日期"
- 时间段冲突: "该时间段已被预约"
- 超过限制: "每天最多只能预约2个器材"
- 器材不足: "该器材在此时间段已全部被预约"
- 缺少教练: "该器材需要教练指导，请先预约同一时间段的教练"
- 权限不足: "无权取消此预约"
- 记录不存在: "预约记录不存在"

### 6.2 错误码设计
- 200: 成功
- 其他: 失败，具体错误信息在msg字段中

## 9. API接口清单

### 9.1 教练预约相关
- `POST /coach` - 预约教练
- `DELETE /coach/{bookingId}` - 取消教练预约
- `GET /coach/list?coachId={id}&date={date}` - 查询教练预约列表
- `GET /coach/my` - 获取我的教练预约
- `GET /coach/all?pageNum={n}&pageSize={n}` - 获取所有教练预约(管理员)

### 9.2 器材预约相关
- `POST /equipment` - 预约器材
- `DELETE /equipment/{bookingId}` - 取消器材预约
- `GET /equipment/list?equipmentId={id}&date={date}` - 查询器材预约列表
- `GET /equipment/my` - 获取我的器材预约
- `GET /equipment/all?pageNum={n}&pageSize={n}` - 获取所有器材预约(管理员)

## 10. 性能优化建议

### 10.1 数据库优化
- 为常用查询字段添加索引(user_id, booking_date, equipment_id/coach_id)
- 定期清理历史预约记录
- 使用分页查询避免大数据量返回

### 10.2 缓存策略
- 可以考虑缓存热门时间段的预约状态
- 缓存器材基本信息减少远程调用

### 10.3 并发控制
- 在高并发场景下考虑使用分布式锁
- 数据库层面使用乐观锁或悲观锁防止超卖
