# 无人健身房管理系统需求分析报告

## 1. 引言

### 1.1 编写目的
本文档旨在详细描述无人健身房管理系统的功能需求和非功能需求，为系统的设计、开发和测试提供依据。

### 1.2 项目背景
随着健身行业的快速发展和人们健康意识的提高，传统健身房的人工管理模式已经难以满足现代用户对便捷性、智能化的需求。无人健身房管理系统通过整合人脸识别、AI智能问答、在线预约等技术，实现健身房的无人化、智能化管理，提升用户体验和运营效率。

### 1.3 术语定义
| 术语 | 说明 |
|------|------|
| JWT | JSON Web Token，用于用户身份认证 |
| AI | Artificial Intelligence，人工智能 |
| NLP | Natural Language Processing，自然语言处理 |
| YOLO | You Only Look Once，目标检测算法 |
| 微服务 | 将应用拆分成多个小型服务的架构模式 |

## 2. 功能性需求

### 2.1 用户管理模块

#### 2.1.1 用户注册
- **需求编号**: FR-001
- **需求描述**: 用户可以通过填写用户名、密码、邮箱等信息进行注册
- **输入**: 用户名、密码、邮箱
- **输出**: 注册成功提示或错误信息
- **前置条件**: 用户名未被注册
- **后置条件**: 用户信息存储到数据库

#### 2.1.2 用户登录
- **需求编号**: FR-002
- **需求描述**: 用户可以通过用户名和密码登录系统，系统使用JWT进行身份认证
- **输入**: 用户名、密码、角色（用户/管理员）
- **输出**: JWT令牌或登录失败提示
- **前置条件**: 用户已注册
- **后置条件**: 生成JWT令牌，记录登录状态

#### 2.1.3 个人信息管理
- **需求编号**: FR-003
- **需求描述**: 用户可以查看和修改个人信息，包括用户名、邮箱、头像等
- **输入**: 修改后的个人信息
- **输出**: 更新成功提示
- **前置条件**: 用户已登录
- **后置条件**: 用户信息更新

#### 2.1.4 会员状态管理
- **需求编号**: FR-004
- **需求描述**: 系统记录用户的会员状态（会员/非会员）和会员到期时间
- **输入**: 订单支付信息
- **输出**: 会员状态更新
- **前置条件**: 用户购买会员套餐
- **后置条件**: 更新会员到期时间

### 2.2 预约管理模块

#### 2.2.1 器材预约
- **需求编号**: FR-005
- **需求描述**: 用户可以查看可预约器材列表，选择合适的器材和时间段进行预约
- **输入**: 器材ID、预约日期、时间段
- **输出**: 预约成功确认或失败原因
- **前置条件**: 用户已登录且为有效会员
- **后置条件**: 生成器材预约记录

#### 2.2.2 教练预约
- **需求编号**: FR-006
- **需求描述**: 用户可以查看可预约教练列表，选择教练和时间段进行预约
- **输入**: 教练ID、预约日期、时间段
- **输出**: 预约成功确认或失败原因
- **前置条件**: 用户已登录且为有效会员
- **后置条件**: 生成教练预约记录

#### 2.2.3 预约查询
- **需求编号**: FR-007
- **需求描述**: 用户可以查看自己的历史预约记录，包括器材预约和教练预约
- **输入**: 预约类型筛选（器材/教练）
- **输出**: 预约记录列表
- **前置条件**: 用户已登录
- **后置条件**: 无

#### 2.2.4 预约取消
- **需求编号**: FR-008
- **需求描述**: 用户可以取消未到期的预约
- **输入**: 预约ID
- **输出**: 取消成功确认
- **前置条件**: 用户已登录，预约未过期
- **后置条件**: 更新预约状态为已取消

### 2.3 订单管理模块

#### 2.3.1 套餐查看
- **需求编号**: FR-009
- **需求描述**: 用户可以查看所有可用的会员套餐，包括套餐名称、价格、等级、时长等信息
- **输入**: 无
- **输出**: 套餐列表
- **前置条件**: 用户已登录
- **后置条件**: 无

#### 2.3.2 套餐购买
- **需求编号**: FR-010
- **需求描述**: 用户可以选择套餐进行购买，生成订单并更新会员状态
- **输入**: 套餐ID
- **输出**: 订单生成确认
- **前置条件**: 用户已登录
- **后置条件**: 生成订单记录，更新会员到期时间

#### 2.3.3 订单续费
- **需求编号**: FR-011
- **需求描述**: 会员到期后，用户可以进行续费操作
- **输入**: 套餐ID
- **输出**: 续费成功确认
- **前置条件**: 会员已过期
- **后置条件**: 更新会员到期时间

### 2.4 人脸识别模块

#### 2.4.1 人脸数据采集
- **需求编号**: FR-012
- **需求描述**: 系统通过摄像头采集用户人脸数据，用于后续识别
- **输入**: 人脸图像
- **输出**: 人脸特征数据
- **前置条件**: 用户已注册
- **后置条件**: 存储人脸特征数据

#### 2.4.2 人脸识别签到
- **需求编号**: FR-013
- **需求描述**: 用户通过人脸识别进行入场签到，系统自动记录到场时间
- **输入**: 人脸图像
- **输出**: 识别结果、签到成功提示
- **前置条件**: 用户已注册且已采集人脸数据
- **后置条件**: 记录入场时间，更新在场状态

#### 2.4.3 人脸识别签退
- **需求编号**: FR-014
- **需求描述**: 用户离场时通过人脸识别进行签退，系统自动记录离场时间
- **输入**: 人脸图像
- **输出**: 识别结果、签退成功提示
- **前置条件**: 用户已在场
- **后置条件**: 记录离场时间，更新在场状态

### 2.5 人数检测模块

#### 2.5.1 实时人数统计
- **需求编号**: FR-015
- **需求描述**: 系统通过摄像头实时监控场馆内人数，检测人员进出

#### 2.5.2 人数异常预警

- **输入**: 当前人数、容量阈值
- **输出**: 预警提示
- **前置条件**: 人数超过容量阈值
- **后置条件**: 无

### 2.6 AI知识库模块

#### 2.6.1 智能问答
- **需求编号**: FR-017
- **需求描述**: 用户可以通过自然语言向AI助手提问，获取健身相关的知识和建议
- **输入**: 自然语言问题
- **输出**: AI回答内容
- **前置条件**: AI服务正常运行
- **后置条件**: 无

#### 2.6.2 问答历史记录
- **需求编号**: FR-018
- **需求描述**: 系统记录用户的问答历史，方便用户查看

- **前置条件**: 用户已登录

### 2.7 数据统计模块

#### 2.7.3 在场人数监控
- **需求编号**: FR-021

- **输入**: 无

## 3. 非功能性需求

### 3.1 性能需求

#### 3.1.1 响应时间
- **需求编号**: NFR-001
- **需求描述**: 系统页面加载时间不超过3秒，API响应时间不超过2秒（95%的请求）
- **指标**: 
  - 页面加载时间 ≤ 3秒
  - API响应时间 ≤ 2秒（P95）

#### 3.1.2 并发处理
- **需求编号**: NFR-002
- **需求描述**: 系统能够支持至少100个并发用户同时操作
- **指标**: 并发用户数 ≥ 100

#### 3.1.3 人脸识别性能
- **需求编号**: NFR-003
- **需求描述**: 人脸识别响应时间不超过1秒，识别准确率不低于95%
- **指标**: 
  - 识别响应时间 ≤ 1秒
  - 识别准确率 ≥ 95%

### 3.2 可用性需求

#### 3.2.1 系统可用性
- **需求编号**: NFR-004
- **需求描述**: 系统可用性不低于99.9%，计划内维护时间除外
- **指标**: 系统可用性 ≥ 99.9%

#### 3.2.2 故障恢复
- **需求编号**: NFR-005
- **需求描述**: 系统在发生故障后能够在5分钟内恢复服务
- **指标**: 故障恢复时间 ≤ 5分钟

### 3.3 安全性需求

#### 3.3.1 身份认证
- **需求编号**: NFR-006
- **需求描述**: 系统使用JWT实现无状态身份认证，令牌有效期为24小时
- **指标**: JWT令牌有效期 = 24小时

#### 3.3.2 数据加密
- **需求编号**: NFR-007
- **需求描述**: 用户密码采用加密存储，敏感数据传输采用HTTPS协议
- **指标**: 
  - 密码加密存储
  - 数据传输采用HTTPS

#### 3.3.3 权限控制
- **需求编号**: NFR-008
- **需求描述**: 系统实现基于角色的访问控制（RBAC），区分用户和管理员权限
- **指标**: 权限控制粒度达到功能级别

### 3.4 可扩展性需求

#### 3.4.1 服务扩展
- **需求编号**: NFR-009
- **需求描述**: 系统采用微服务架构，支持服务的水平扩展
- **指标**: 支持服务实例动态扩缩容

#### 3.4.2 功能扩展
- **需求编号**: NFR-010
- **需求描述**: 系统架构支持新功能的快速开发和集成
- **指标**: 新功能开发周期 ≤ 2周

### 3.5 兼容性需求

#### 3.5.1 浏览器兼容
- **需求编号**: NFR-011
- **需求描述**: 前端页面兼容主流浏览器（Chrome、Firefox、Edge、Safari）的最新两个版本
- **指标**: 支持Chrome、Firefox、Edge、Safari最新两个版本

#### 3.5.2 设备兼容
- **需求编号**: NFR-012
- **需求描述**: 前端页面支持响应式设计，适配桌面端和移动端设备
- **指标**: 支持屏幕宽度 ≥ 375px的设备

### 3.6 易用性需求

#### 3.6.1 用户界面
- **需求编号**: NFR-013
- **需求描述**: 界面设计简洁直观，用户能够在无培训情况下完成基本操作
- **指标**: 新用户完成首次预约操作时间 ≤ 5分钟

#### 3.6.2 操作反馈
- **需求编号**: NFR-014
- **需求描述**: 系统对用户操作提供即时反馈，包括成功提示、错误提示等
- **指标**: 操作反馈延迟 ≤ 1秒

### 3.7 可维护性需求

#### 3.7.1 代码规范
- **需求编号**: NFR-015
- **需求描述**: 代码遵循统一的编码规范，包含必要的注释
- **指标**: 代码注释覆盖率 ≥ 20%

#### 3.7.2 日志记录
- **需求编号**: NFR-016
- **需求描述**: 系统记录关键操作日志和异常日志，便于问题排查
- **指标**: 关键操作100%记录日志
